# OB_backup

## PL经历
负责 OceanBase数据库中 PL缓存子系统 的设计、开发与维护，涵盖两个核心子模块：  

● PL 内存缓存（PL Cache）：基于 Lib Cache 统一框架，缓存编译后的存储过程、函数、包、触发器、匿名块等 PL 对象，避免重复编译(parse+resolve+code generate)  

● PL 磁盘缓存（PL Disk Cache）：将 LLVM JIT 编译生成的二进制（DLL）持久化到内部表，重启/跨节点复用编译结果，避免LLVM重编译(code generate)  

两层缓存协同工作：执行时先查内存缓存，未命中则经过parse+resolve后，再查磁盘缓存加载预编译二进制，均未命中才触发完整的LLVM编译流程。
该模块直接影响存储过程的用户体验，尤其是体量大的package，重新编译耗时很长，会引起业务抖动。

我做了哪些事情呢？

如何判断过期？

1.记录依赖对象id及对应schema version，缓存匹配时重新捞一遍依赖对象schema，如果schema version升高了，那么该缓存对象需要过期处理。

2.对于表对象，可能有alter table rename / truncate table 等不会改变表结构的ddl操作，维护依赖表列信息，在表依赖schema version升高时，判断表列信息是否有变化。

pl编译链路，parse + resolve + code generate 。 

LLVM 编译完产物后，直接存入磁盘缓存。如何判断过期？

内存级缓存是怎么比较的？存了所有依赖的schema version，基于新旧依赖schema version，判断依赖是否做了ddl。

磁盘缓存做的比较简单，

1.存入时只记录当前租户的schema version，叫做merge version

2.查缓存时，因为做过resolve，能直接拿到所有依赖，也捞一份schema version，分别和merge version比较。对于表依赖，有一个优化，如果truncate 表导致的schema version升高，比较表列信息是否变化。这里通过收集依赖对象schema id+表依赖列信息，然后MD5 序列化成一个字符串。用来做等值匹配。

###  pl异步提交
蚂蚁内部有一个账务系统，小机器规格（6c48G集群）需要支持峰值10w tps，优化后从8w -> 9w tps，测试场景是固定的。 同步阻塞时，pl工作线程在空闲等待，内存也不能及时释放，资源利用不够高效。

事务层有一个异步任务队列，提交时可以绑定callback。

原本pl内部的commit，都是同步提交，执行到commit时，sql线程会阻塞等待事务commit结束，并返回结果后，才能处理后序sql语句。本地事务大部分commit在微妙级，分布式事务commit在毫秒级，阻塞等待不仅会占用sql连接，而且会提高延迟，体感很差。

在热点行场景，大量请求打到数据库端，sql工作线程由于以上阻塞，达不到业务要求的吞吐，因此尝试改造异步提交以提高系统的吞吐量。

之前我们支持sql的异步回包，事务层也封装了异步事务提交的接口。sql在适配异步回包时，基本只需要封一个callback对象，然后交给事务层，这时sql直接返回，后面释放掉该连接。

在事务结束commit时，会执行该callback函数，完成给客户端回包，清理session等动作。

对于PL来说，一次请求内部，是可能有多个commit的，和sql的区别是，pl的最后一个commit才需要回包，中间的所有commit都不能回包。

这是第一个问题，如何找到pl层的最后一个commit，编译期能做到这件事吗？不能，只有在执行期，才能找到最后一个commit，然后我们标记下这个commit，让他回包。

这里就涉及到 sql线程 和 事务 线程 如何进行 状态同步了。在callback 对象上用一把锁进行控制，避免读写冲突。pl执行结束时，尝试拿这把锁，能拿到的话，检查callback是否执行过了，没有执行过，标记需要回包。如果已经执行过了，说明事务已经结束了，那么还是有sql线程回包。

还有一个问题，事务层的异步对列，不保证按照提交事务的顺序执行。因此需要在pl层维持这个顺序。目前的策略是，pl后序的事务commit前，要等前序事务结束。这里其实还有一个语义问题，改造成异步commit后，pl后面执行的sql语句，能不能看到前一个事务修改过的数据呢？本地事务开启ELR优化，提前释放锁可以，但是分布式场景不能释放锁，select for update 是可能读不到数据的。这和原来pl 同步提交的语义不一致。

目前的策略是，事务层收集一个事务会修改的表，pl在执行后序sql前，先从计划上判断本条sql是否依赖前序事务修改过的表，如果有，就等待，直到前序事务执行结束。

错误处理这块，异步事务执行失败，会同步状态给pl，pl提交后面的事务之前，检查到前序事务失败，先回滚掉当前事务，在把错误码抛出去。
